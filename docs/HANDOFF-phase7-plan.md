## 향후 개선 과제 (우선순위 순)

### 1. 본문 크롤링 개선
- **현상**: Google News URL 디코딩은 해결했으나, 일부 사이트(JS 렌더링)는 여전히 크롤링 실패
- **방안**: Playwright/Puppeteer headless browser 도입
- **기대효과**: Top 5 기사 본문 크롤링률 50% → 90%+

### 2. 카테고리별 요약 심화
- **현상**: Top 5만 Fact/Impact/Risk/Next 구조, 나머지는 headline+fact만
- **방안**: Stage 3에서도 Impact 필드 추가, 또는 중요도 높은 카테고리만 심층 분석
- **기대효과**: 브리핑 정보 밀도 향상

### 3. 뉴스 수집 튜닝
- **현상**: 키워드 매칭 기반 스코어링으로 노이즈 존재
- **방안**:
  - LLM 기반 관련도 판정 (수집 후 1차 필터에 경량 모델 적용)
  - 키워드 세트 주기적 업데이트
  - 소스별 가중치 조정
- **기대효과**: 관련 기사 정밀도 향상, 노이즈 감소

### 4. 사용자 피드백 채팅 기능 (향후)
- **목적**: 사용자가 브리핑 결과에 대해 피드백을 주면 프로필/스코어링에 반영
- **설계 방향**:
  - 브리핑 페이지 내 간단한 채팅 UI
  - "이 기사는 관련 없음", "이 토픽 더 보고 싶음" 등 구조화된 피드백
  - 피드백 → `stable_profile.json`의 가중치 미세 조정 (conversation_hint_policy 활용)
  - 대화 컨텍스트는 5~10% 가중치 제한 유지 (편향 방지)
- **구현 단계**:
  1. 피드백 데이터 모델 정의 (FeedbackEntry)
  2. 피드백 저장소 (SQLite 또는 JSON)
  3. 피드백 → 프로필 반영 로직
  4. 브리핑 페이지 내 채팅 UI (웹소켓 또는 REST)
  5. 피드백 기반 스코어링 가중치 조정

### 5. 검색용 채팅 기능 (향후)
- **목적**: 과거 브리핑 데이터를 검색하고 질문에 답변
- **설계 방향**:
  - "지난주 반도체 수주 관련 뉴스 보여줘" 같은 자연어 질의
  - 저장된 브리핑 JSON을 검색 소스로 활용
  - RAG 파이프라인: 질의 → 관련 브리핑/기사 검색 → LLM 답변 생성
- **구현 단계**:
  1. 브리핑 아카이브 인덱싱 (날짜/카테고리/키워드)
  2. 검색 API (키워드 + 의미 검색)
  3. 채팅 UI (웹 기반)
  4. LLM 답변 생성 (컨텍스트 = 검색된 브리핑 데이터)

### 6. HTML 테마 재설계
- **현상**: 동적 테마(콘텐츠 기반 색상 변경)가 사용자 의도와 다름
- **방안**: 고정 테마 + 날짜/요일별 미세 변경, 또는 사용자 설정 가능
- **우선순위**: 낮음 (기능에 영향 없음)
